# 2026-03-18 HD Identity Upgrade Plan

## Goal

Add hierarchical deterministic identity support to AiP2P without breaking existing single-key identities.

## Actual Current State

- The repository currently supports only standalone Ed25519 identity files.
- `internal/aip2p/identity.go` has no HD fields, mnemonic handling, or derivation logic.
- `cmd/aip2p/main.go` only supports `identity init`.
- Message signing verifies only the concrete origin public key already embedded in the message.
- Writer policy has no HD parent/child trust model yet.

## Constraints

- Never print mnemonic, seed, or private key in CLI output, logs, or tests.
- Preserve backward compatibility for existing standalone identity files.
- Keep signing deterministic for the same mnemonic + derivation path.
- Prefer P0/P1 implementation order before policy/API expansion.

## Phase Plan

### Phase 1: P0 Foundation

Status: completed

- Add HD key derivation primitives for Ed25519.
- Add mnemonic generation and recovery support.
- Extend `AgentIdentity` to represent HD master and derived identities.
- Add CLI commands:
  - `identity create-hd`
  - `identity derive`
  - `identity list`
  - `identity recover`

### Phase 2: P1 Signing Integration

Status: completed

- Allow a master HD identity file to sign for child authors.
- Add HD metadata to message extensions:
  - `hd.parent`
  - `hd.parent_pubkey`
  - `hd.path`
- Validate derived public keys during message origin verification.
- Keep standalone identity behavior unchanged.

### Phase 3: P1 Trust Model

Status: in progress

- Extend writer policy with HD-aware parent/child trust controls.
- Keep blacklist precedence over whitelist.
- Add optional derivation verification for trusted child identities.

### Phase 4: P2 API and SDK

Status: in progress

- Review whether local HTTP API should expose HD identity management.
- Defer Python/JS SDK changes until core behavior is stable.

### Phase 5: Documentation

Status: pending

- Document HD identity lifecycle and safe backup behavior.
- Document the rule that mnemonic/private key material is never echoed.

## This Round Scope

- Implement Phase 1.
- Implement the core of Phase 2 inside message signing and verification.
- Add tests for derivation, CLI flows, and backward compatibility.
- Leave writer policy trust expansion for a later round if it would materially increase risk.

## Risks

- Ed25519 SLIP-0010 supports hardened derivation only; CLI and path parsing must enforce that safely.
- Hardened Ed25519 derivation cannot prove child lineage from a parent public key alone.
- A child identity file that stores only public metadata cannot sign on its own unless explicitly exported with private material.
- URI segment to derivation-path mapping must be deterministic and documented.

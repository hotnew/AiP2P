架构层面

2 项
▶ 1.1 store.go 单文件 3,462 行，职责过多
P2
所有数据模型定义、全部 CRUD、文件锁、索引管理、归档、事件、Webhook、normalize 函数都堆在同一个文件中。
架构可维护性

问题分析

不同关注点耦合在一起，修改任一功能都需要在 3,000+ 行文件中定位代码。新开发者理解成本极高。

建议拆分方案

team/
  types.go           // 所有 struct 定义
  store.go           // Store 结构体 + OpenStore (~200 行)
  team_crud.go       // LoadTeam, ListTeams
  member_crud.go     // LoadMembers, SaveMembers
  policy_crud.go     // LoadPolicy, SavePolicy
  channel_crud.go    // Channel 相关操作
  message_crud.go    // 消息读写 + 分片
  task_crud.go       // 任务 CRUD + 索引/遗留双轨
  artifact_crud.go   // 产物 CRUD
  history_crud.go    // 变更历史
  archive.go         // 归档快照
  index.go           // 索引管理 + 迁移 + 压缩
  event.go           // Subscribe/Publish/Webhook
  normalize.go       // 所有 normalize 函数
  helpers.go         // ID 构建 + 工具函数
▶ 1.2 handler.go 1,300+ 行，缺少结构化组织
P2
40+ 个 handler 函数全部为顶层函数，每个都需手动传入 app 和 store 参数。
架构可维护性

优化方案

引入 TeamHandler 结构体统一持有依赖；按功能拆分为 handler_team.go、handler_task.go、handler_channel.go 等。

type TeamHandler struct {
    app   *newsplugin.App
    store *teamcore.Store
}

func (h *TeamHandler) HandleTeamIndex(w http.ResponseWriter, r *http.Request) {
    // 使用 h.app 和 h.store，无需参数传递
}

性能层面

5 项
▶ 2.1 readLastJSONLLines 逐字节读取 — 性能灾难
P0
每读 1 字节触发 1 次系统调用。10MB 文件需要 ~1000 万次 ReadAt 调用。
性能严重

现状代码（store.go 第 841-879 行）

for pos := info.Size() - 1; pos >= 0; pos-- {
    var b [1]byte
    file.ReadAt(b[:], pos)  // 每字节一次系统调用！
    // ...
}

优化方案：块状反向读取（8KB 块）

一次读取 8KB 数据块，在内存中反向扫描换行符。系统调用从数百万次降至约 1,200 次。

const blockSize = 8192
for offset := size; offset > 0 && len(lines) < limit; {
    readSize := min(blockSize, offset)
    offset -= readSize
    buf := make([]byte, readSize)
    file.ReadAt(buf, offset)  // 一次读 8KB
    // 内存中扫描换行符...
}

预期收益：100x+ 速度提升

▶ 2.2 channelSummary 加载全部消息只为统计数量
P0
LoadMessages(teamID, channelID, 0) 加载整个频道的全部消息，只为获取 len() 和最新时间。
性能严重

影响范围

ListChannels 会对每个频道都调用 channelSummary。如果一个团队有 5 个频道、每个频道 1 万条消息，就需要将 5 万条消息全部加载到内存。

优化方案

方案 A：新增 channelMessageStats 轻量方法，只统计行数 + 读取最后一行获取时间。
方案 B（推荐）：在频道元数据中维护 message_count 和 last_message_at，每次 AppendMessage 时增量更新。

▶ 2.3 LoadTaskMessages 全量扫描所有频道所有消息
P1
查找与某 Task 相关的消息时，遍历所有频道的所有消息做 StructuredData 匹配。
性能

优化方案

利用 Task 的 ChannelID 优先搜索对应频道；或新增按 task_id 的消息索引文件。LoadMessagesByContext 同理。

▶ 2.4 ListTeams 串行加载每个团队信息
P2
50 个团队串行读取 team.json + members.json + channels 配置，I/O 累加。
性能

优化方案

使用 goroutine 并发加载各团队数据，通过 channel 收集结果。

▶ 2.5 handleTeam 错误处理：errOnce 丢弃后续错误
P2
9 个并发 goroutine 加载数据，errOnce 只保留第一个错误，无法提前取消。
性能错误处理

优化方案

使用 golang.org/x/sync/errgroup 替换手动 WaitGroup + errOnce，获得 context 取消支持。

数据一致性与可靠性

3 项
▶ 3.1 JSONL Append 写入不安全 + 错误静默丢弃
P0
程序崩溃可导致半行 JSON；json.Unmarshal 失败时 continue 丢弃数据无日志。
严重可靠性

问题位置

store.go 中约 10 处 json.Unmarshal + continue（第 683、697、746、813、1159、1475 行等）。

修复方案

1. file.Close() 前调用 file.Sync()
2. 所有 continue 处改为 log.Printf("[team] corrupt JSONL line: %v", err)

▶ 3.2 文件锁（Flock）无超时机制
P1
syscall.Flock(LOCK_EX) 阻塞等待，无超时。异常退出可能导致永久阻塞。
可靠性

优化方案

改用 LOCK_NB 非阻塞模式 + 重试循环 + 超时退出（建议 10 秒）。

▶ 3.3 Webhook 推送缺少重试和日志
P1
所有错误静默丢弃，无重试机制，每次创建新 http.Client 实例。
可靠性

优化方案

复用 http.Client；增加最多 3 次重试 + 指数退避；区分 500/429（可重试）和 4xx（不可重试）；添加日志。

▶ 4.4 mergeChannel 的 CreatedAt 逻辑 Bug
P0

Bug

Bug 代码

if base.CreatedAt.IsZero() {
    base.CreatedAt = override.CreatedAt  // 第一段
}
if override.CreatedAt.IsZero() {
    // keep existing
} else {
    base.CreatedAt = override.CreatedAt  // 又覆盖了！

    base.CreatedAt = override.CreatedAt

安全性

2 项
▶ 5.1 teamRequestTrusted IP 校验可被反向代理绕过
P1
在 nginx 等反向代理后面，RemoteAddr 可能是代理 IP 而非真实客户端 IP。
安全

建议

明确不信任 X-Forwarded-For 头；生产环境建议增加 token-based auth。

▶ 5.2 NormalizeTeamID 缺少路径遍历防护
P1
虽然 / 被替换为 -，但未处理 URL 编码 %2F、.. 等形式。
安全

修复要点

1. 先做 URL 解码 url.PathUnescape
2. 过滤 ..
3. 最终检查 filepath.IsAbs

接口 / 测试 / 可观测性 / 同步

7 项
▶ 6.1 limit=0 语义不明确
P3
LoadMessages(teamID, channelID, 0) 的 0 表示"加载全部"，容易误读为"不加载"。
API 设计

方案

引入 const LoadAll = -1 明确语义，或新增 LoadAllMessages 方法。

▶ 6.2 全部方法缺少 context.Context
P3
无法传递 HTTP 请求 deadline/timeout，无法做分布式追踪。
架构

方案

长期为所有 public 方法添加 ctx 参数；短期先为 withTeamLock 增加 context 支持。

▶ 7.1 store.go（核心数据层）完全没有单元测试
P3
team_sync_test.go 有 1,065 行测试，但 store.go 零测试覆盖。
测试

建议覆盖

readLastJSONLLines 边界、索引存储 CRUD、文件锁并发、Policy.Allows、NormalizeTeamID 等至少 20+ 测试用例。

▶ 8.1 整个 team 包零日志记录
P1
JSON 反序列化失败、Webhook 失败、锁超时等全部静默处理。
可观测性

方案

引入 log/slog，在 Store 中持有 Logger，关键位置输出结构化日志。

▶ 8.2 缺少 metrics 指标
P3
无法监控消息吞吐、任务数、Webhook 错误率、锁等待时间等关键指标。
可观测性

方案

使用 expvar 或 Prometheus client 收集关键指标。

▶ 9.1 Key() 方法重复 normalize
P3
sync.go 的 Key() 方法对 Type 再次做 ToLower+TrimSpace，Normalize() 已经做过。
代码质量

方案

Key() 内部先调用 Normalize()，或要求调用方确保已 normalized。

▶ 9.2 时间戳冲突检测在时钟偏移下不可靠
P3
分布式节点间的 UpdatedAt 比较依赖时钟同步，NTP 偏移可导致错误的 last-write-wins。
架构

长期方案

引入 Hybrid Logical Clock (HLC) 替代纯时间戳。
